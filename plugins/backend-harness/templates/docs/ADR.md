# ADR (Architecture Decision Records)

> **append-only**. 기존 항목은 수정하지 않는다 — 결정이 뒤집혔으면 새 ADR을 추가해 이전 항목을
> "Superseded by ADR-N"으로 표시한다. 번호 충돌은 머지 시 조정한다.
> `CLAUDE.md`의 "협업 규칙" 섹션 참조.

## 작성 형식

```
## ADR-{N}: {결정 제목}

- **날짜**: YYYY-MM-DD
- **상태**: Accepted | Superseded by ADR-{M}
- **배경**: 어떤 문제/제약 때문에 이 결정이 필요했는가
- **결정**: 무엇을 하기로 했는가
- **대안**: 검토했던 다른 선택지와 기각 이유
- **결과**: 이 결정으로 감수하는 트레이드오프
```

## ADR-1: 아키텍처 스타일 — 모듈러 모놀리스 + 선택적 헥사고날/DDD

- **날짜**: 2026-07-08
- **상태**: Accepted
- **배경**: 파편화된 백엔드 서버들을 하나로 통합하면서, 향후 트래픽·팀 규모에 따라 일부
  모듈을 MSA로 분리하기 쉬운 구조가 필요하다. 단, 현재 팀 규모에서 전 모듈 풀 헥사고날 DDD의
  보일러플레이트 비용(모듈마다 port/adapter/매핑 레이어)은 회수가 불확실하다.
- **결정**:
  1. **모듈러 모놀리스** — 단일 배포 단위, 모듈 구성은 `ARCHITECTURE.md` "모듈/서비스 지도" 참조.
  2. **Spring Modulith 도입** — 모듈 경계를 문서·리뷰가 아니라 `ApplicationModules.verify()`
     테스트로 CI 수준에서 강제한다. 모듈 간 비동기 통신은 Event Publication Registry 사용.
  3. **DDD는 전략적 설계만 전면 적용** — 모듈 = Bounded Context, 모듈 간 통신은 공개
     인터페이스/도메인 이벤트만.
  4. **헥사고날·DDD 전술 패턴은 모듈별 선택 적용** — `order`(Shopify port/adapter + 도메인 모델),
     `message`(발송 채널 port/adapter)부터. 나머지 모듈은 표준 레이어드로 시작하고 도메인
     복잡도가 실제로 증가하면 승격한다. 적용 수준 표는 `.claude/rules/layer-architecture.md` 참조.
  5. **모듈별 DB 소유권** — 테이블은 단일 모듈 소유, 교차 모듈 쿼리/조인 금지.
- **대안**:
  - 전 모듈 풀 헥사고날 DDD — MSA 분리 용이성의 대부분은 모듈 경계·이벤트·DB 분리에서 나오며,
    내부 레이어링의 기여는 작다. CRUD 성격 모듈(community, file 등)에서 비용 회수 불가로 기각.
  - Vertical Slice Architecture — Spring 생태계 관례·도구 지원 부족, 온보딩 비용으로 기각.
  - 처음부터 MSA — 서버 통합(운영 부담 축소) 목적과 정면 충돌로 기각.
- **결과**: 모듈별 아키텍처 스타일이 이원화된다(레이어드 vs 헥사고날) — 적용 수준 표를 규칙
  파일에 명시해 혼선을 막는다. 레이어드 모듈을 나중에 헥사고날로 승격하는 리팩토링 비용은
  감수한다(모듈 경계가 유지되는 한 크지 않다).
