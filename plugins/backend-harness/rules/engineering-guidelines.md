# 엔지니어링 행동 규범

특정 기술 스택 규칙이 아니라 **에이전트가 코드를 짤 때 지켜야 할 행동 습관**이다.
참고: Andrej Karpathy의 LLM 코딩 실수 관찰(karpathy-guidelines).

**Tradeoff**: 아래 원칙은 속도보다 신중함을 우선한다. 사소한 작업(오탈자 수정 등)에는
판단력을 발휘해 과도하게 적용하지 않는다.

## 1. 코딩 전에 생각하라 — 가정을 표면화하라

- 요청에 불확실한 부분이 있으면 가정을 명시하고 진행한다 ("~라고 가정하고 진행합니다").
  판단이 서지 않으면 진행하지 말고 사용자에게 되묻는다.
- 해석이 여러 개 가능하면 침묵하고 하나를 고르지 말고, Plan에 대안으로 함께 제시한다.
- 더 단순한 대안이 있다고 판단되면 이유와 함께 제안한다 (요청받은 방식이 최선이 아닐 수 있다는
  것을 침묵하고 넘어가지 않는다).
- `api-developer` Phase 1 Plan에는 이 항목을 "가정 및 대안" 섹션으로 명시한다
  (backend-harness 플러그인의 `api-developer` 에이전트 정의 참조).

## 2. 단순함을 기본값으로 — 요청 이상을 만들지 않는다

- 요청받지 않은 기능·설정 가능성·유연성을 추가하지 않는다.
- 발생 가능성이 없는 시나리오에 대한 방어 코드를 추가하지 않는다.
- 단일 사용처 코드에 추상화 계층을 만들지 않는다.
- **예외**: `MESSAGE_BROKER: application-event` 설정일 때 `ops-checker.md`가 권고하는
  `EventPublisher` 추상화는 실제 운영 설정(향후 Kafka 전환 계획이 있다는 신호)에 근거한 것이므로
  이 원칙의 예외로 허용한다. 이 예외 외에 "나중에 필요할 수도 있어서" 식의 설계는 허용하지 않는다.
- 스스로에게 물어라: "시니어 엔지니어가 보면 과설계라고 할까?" 그렇다면 단순화한다.

## 3. 수술하듯 변경하라 — 지정된 범위만 건드린다

- 기존 코드를 편집할 때 요청받지 않은 인접 코드·주석·포맷을 "개선"하지 않는다.
- 깨지지 않은 것을 리팩토링하지 않는다.
- 기존 스타일이 마음에 들지 않아도 그대로 맞춘다(Checkstyle 기준 준수 범위 내에서).
- 무관한 dead code를 발견하면 삭제하지 말고 언급만 한다.
- 자신의 변경으로 인해 생긴 미사용 import/변수/함수만 정리한다. 기존에 있던 미사용 코드는
  요청받지 않는 한 그대로 둔다.
- **검토-수정 사이클(fix 담당)에서 특히 중요**: `FOCUS`에 명시된 이슈만 수정한다. 수정 과정에서
  무관한 문제를 발견하면 고치지 말고 `ISSUES`에 별도 항목으로 보고만 한다
  (backend-harness 플러그인의 `harness-review-cycle` 스킬 참조).
- 판별 기준: 변경한 모든 줄이 사용자 요청으로 직접 추적 가능해야 한다.

## 4. 목표 중심으로 실행하라 — 검증 기준을 정의하고 그것으로 판단하라

- 작업을 검증 가능한 목표로 변환한다:
  - "유효성 검사 추가" → "잘못된 입력에 대한 테스트를 작성하고 통과시켜라"
  - "버그 고쳐" → "재현하는 테스트를 먼저 작성한 뒤 통과시켜라" (버그 수정 요청은
    backend-harness 플러그인의 `harness-bugfix` 스킬이 정한 재현-우선 순서를 따른다)
  - "X 리팩토링" → "리팩토링 전후 테스트가 모두 통과하는지 확인하라"
- 다단계 작업은 Plan에 아래 형식으로 단계별 검증 기준을 명시한다:
  ```
  1. [단계] → verify: [확인 방법]
  2. [단계] → verify: [확인 방법]
  ```
- 검증 기준이 강할수록 사용자 개입 없이 독립적으로 사이클을 돌 수 있다. "동작하게 만들어라" 같은
  약한 기준은 매번 사용자 확인이 필요해진다.
